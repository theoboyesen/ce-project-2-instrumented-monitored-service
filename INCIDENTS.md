# Incident Reports

This document describes simulated production incidents used to validate the observability and monitoring capabilities of the Order API service.

The incidents demonstrate how metrics, dashboards, alarms, and logs can be used to detect, investigate, and resolve system issues.

---

# Incident 1: High CPU Utilization

## Summary
A CPU saturation event was intentionally triggered on the EC2 instance hosting the Order API service. The monitoring system successfully detected the anomaly and triggered a CloudWatch alarm.

## Detection
The incident was detected by the CloudWatch alarm:

cpu-utilization-high

The alarm triggers when:

CPUUtilization > 80%  
for 2 consecutive evaluation periods (60 seconds each).

The alarm transitioned from:

OK → ALARM

An email notification was sent via the SNS topic `order-api-alerts`.

## Investigation

The CloudWatch monitoring dashboard was used to analyze system behaviour.

Observed metrics:

CPU Utilization spiked to approximately 99%  
Memory utilization remained stable  
Disk usage remained stable  
API latency and error rate were unchanged  

This indicated that the issue was related specifically to **compute saturation**, rather than application failure.

## Root Cause

The high CPU utilization was caused by intentionally running CPU stress processes on the EC2 instance:


yes > /dev/null &
yes > /dev/null &


These commands generated continuous CPU load on both vCPUs of the `t3.micro` instance.

## Resolution

The CPU stress processes were terminated using:


pkill yes


CPU utilization returned to normal levels and the CloudWatch alarm automatically returned to the **OK** state.

## Lessons Learned

The monitoring system successfully detected infrastructure resource saturation and generated alerts.

This demonstrates that the observability system can identify infrastructure performance issues in production environments.

# Incident 2: API Error Spike

## Summary
An application-level error spike was simulated by sending repeated invalid requests to the API. The monitoring system detected the abnormal error rate and triggered an alert.

## Detection

The incident was detected by the CloudWatch alarm:

api-errors-high

The alarm triggers when:

api_errors_total > 5 within 60 seconds

The alarm transitioned from:

OK → ALARM

An alert notification was sent via SNS.

## Investigation

The CloudWatch dashboard showed a sudden spike in the **API Errors** metric while other metrics remained stable.

Observed metrics:

API Errors increased rapidly  
CPU utilization remained normal  
Memory utilization remained normal  
API latency remained stable  

This suggested the issue was related to **application behaviour rather than infrastructure resources**.

Logs in CloudWatch Logs confirmed multiple requests to an invalid endpoint.

Example request:


GET /orders/invalid


## Root Cause

The error spike was intentionally generated by sending multiple requests to a non-existent order ID:


for i in {1..20}; do curl http://localhost:5000/orders/invalid
; done


This resulted in repeated **404 responses**, which were logged as `order_not_found` events and converted into the `api_errors_total` metric.

## Resolution

Once the invalid requests stopped, the error rate returned to normal levels and the CloudWatch alarm automatically returned to the **OK** state.

## Lessons Learned

The monitoring system successfully detected abnormal application behaviour and alerted operators.

This demonstrates that the system can detect and notify operators about application-level failures in addition to infrastructure issues.
